# 原始笔记入库与组织规范

## 适用范围
所有待进入 `raw/` 文件夹的新笔记，以 Markdown 文件为主。

## 核心原则
1. **原文不动**：笔记正文内容、排版、链接一律保留原样，不做实质性修改。
2. **图片base64嵌入**：所有图片必须转换为base64编码直接嵌入markdown文件中，不保留外部图片引用。
3. **先筛后存**：入库前必须通过基本质量检查，无意义内容不进入知识库。
4. **分类清晰**：按主题层级归档，命名规范统一，方便人直接浏览和后续脚本处理。
5. **入库即提炼**：笔记进入`raw/`正式目录时，同时完成提炼到`core/`，节省token消耗。
6. **tmp为暂存区**：`raw/tmp/`用于存放零散笔记，提炼时忽略；只有进入`raw/`其他文件夹的笔记才进行提炼。
7. **先同步再整理**：每次整理笔记前，必须先执行 `git pull` 拉取GitHub上的最新版本，确保本地知识库与远程同步，避免覆盖他人修改或产生冲突。

## 一、入库前检查

### 1.1 内容有效性判断
打开文件，快速判断内容是否合格。出现以下情况视为**不合格**，应拒绝入库或移至废纸篓：
- 正文为空或仅有标题
- 内容为纯乱码、无法辨认的文字
- 整篇为无意义的测试文本（如"asdfg"、"测试测试"）
- 与知识库定位完全无关的临时信息（如外卖订单、一次性的验证码）

### 1.2 结构严谨性判断
内容合格的前提下，进一步判断结构是否**可阅读、可提炼**：
- **合格标准**：有明确的主题或观点，段落基本分明，能看出表达意图
- **轻微瑕疵可接受**：偶尔的错别字、多余空行、标点不规范等，可在入库时顺手修正，但不做深度润色
- **结构混乱需标记**：如果内容本身有价值但排版极其混乱（如无任何换行、标题层级乱用），仍可入库，但应在日志中备注"需后续整理"，并考虑在提炼阶段重点处理

### 1.3 筛查结论
- **通过** → 进入下一步：命名与分类
- **不通过** → 删除或移至 `raw/_archived/`，记录日志
- **存疑** → 保留入库，日志中标记 `REVIEW`，待人工二次判断

## 二、文件命名规范

### 2.1 命名格式
```
序号_标题.md
```
- **序号**：两位数字（如 `01`、`02`），表示在该主题文件夹内的推荐阅读顺序
- **分隔符**：下划线 `_`
- **标题**：用中文或英文简洁概括笔记核心内容，避免特殊字符（`/ \ : * ? " < > |`）

### 2.2 序号规则
- **序号含义**：表示在该主题文件夹内的**推荐阅读顺序**，从基础到进阶
- **新文件递补**：查看目标文件夹中已有文件的最大序号，新文件序号递补
- **序号复用**：如果某个序号被删除，新笔记可以复用该序号，保持连续性
- **重新编号**：如需调整阅读顺序，允许重新编号已有文件，但需在日志中记录所有改名操作
- **序号连续性**：同一文件夹内序号应连续（01、02、03...），避免跳跃（01、02、08...）

### 2.2.1 序号排序原则
当需要重新编号时，按以下逻辑排序：
1. **基础概念**在前，**进阶应用**在后
2. **理论知识**在前，**实践实现**在后
3. **单点知识**在前，**综合案例**在后
4. **独立模块**在前，**依赖模块**在后

### 2.2.2 序号排序示例
```
✅ 推荐顺序：
01_协议基础与原理.md      ← 基础概念
02_底层实现方法.md        ← 底层实现
03_接口层操作A.md         ← 接口层单点
04_接口层操作B.md         ← 接口层进阶
05_综合测试.md            ← 最终验证
06_完整案例.md            ← 综合案例（可选）

❌ 避免的顺序：
01_协议基础.md
02_底层实现.md
08_接口层操作.md          ← 序号跳跃
10_测试.md
```

### 2.3 标题提炼规则
- **有现成标题**：沿用原标题，去除多余的修饰词和特殊符号
- **无标题或标题无意义**：根据全文内容总结一个 3~8 个字的标题
- **标题语言**：优先使用笔记正文的主要语言；中英文混合的主题，中文优先

### 2.4 命名示例
```
✅ 01_决策树调参经验.md
✅ 02_过拟合与正则化.md
✅ 03_Transformer架构图解.md

❌ 笔记1.md              （标题无意义）
❌ 01_笔记.md             （标题太笼统）
❌ 02_机器学习/深度学习.md （含非法字符 /）
```

## 三、主题分类规范

### 3.1 主题树结构
`raw/` 下按主题层级建立文件夹树，从宽泛到具体逐步深入：
```
raw/
└─ 一级主题/
   └─ 二级主题/
      └─ 三级主题/
         └─ 四级主题/
            └─ 笔记文件.md
```

### 3.2 层级限制
- **最少 1 层**（至少有一个一级主题文件夹）
- **最多 5 层**（一级到五级），超过 5 层说明分类过细，需要合并或重新划分
- 如果某个主题下笔记很少（≤3 篇），暂缓建立子层级，等数量积累后再细分

### 3.3 文件夹命名
- 使用**中文主题词**，不加序号
- 命名应能准确涵盖该文件夹下所有笔记的共性
- 同一层级的文件夹之间互斥，避免内容重叠
- 示例：
  ```
  ✅ 计算机科学/人工智能/机器学习
  ✅ 个人成长/时间管理
  ✅ 项目管理/敏捷开发/Scrum实践

  ❌ 学习/编程/编程语言    （概念重合）
  ❌ 01-技术/02-生活       （不需要序号）
  ```

### 3.4 归类判断流程
1. 阅读笔记，提取核心主题词
2. 在已有主题树中查找最匹配的路径
3. 如果匹配多个可能路径，选择**最直接、最具体**的那个
4. 如果现有主题都不匹配，评估是否需要新建主题文件夹：
   - 预计该主题后续还会有 ≥3 篇相关笔记 → 新建
   - 只是孤立的单篇 → 暂放入相近主题，或用 `_待归类/` 临时存放
   - 也可使用 `raw/tmp/` 作为临时存放点，待处理后再归入正式分类

## 四、分类存放示例

假设当前 `raw/` 结构如下：
```
raw/
├── 计算机科学/
│   └── 编程语言/
│       ├── 01_Python装饰器详解.md
│       └── 02_Rust所有权机制.md
├── 个人成长/
│   └── 阅读笔记/
│       └── 01_如何阅读一本书.md
```

### 场景一：有匹配路径
收到一篇关于"Python 协程"的笔记。
- 归属路径：`计算机科学/编程语言/`
- 已有文件序号最大为 `02`，新文件命名为 `03_Python协程异步编程.md`
- 放入后：
  ```
  raw/
  └── 计算机科学/
      └── 编程语言/
          ├── 01_Python装饰器详解.md
          ├── 02_Rust所有权机制.md
          └── 03_Python协程异步编程.md
  ```

### 场景二：需要新建层级
收到一篇关于"决策树算法"的笔记。
- 当前无"机器学习"相关路径，但属于"计算机科学"分支
- 判断该主题后续还会产出笔记 → 新建子文件夹
- 创建路径：`计算机科学/机器学习/`
- 命名文件：`01_决策树基础概念.md`
- 放入后：
  ```
  raw/
  └── 计算机科学/
      ├── 编程语言/
      │   └── ...
      └── 机器学习/
          └── 01_决策树基础概念.md
  ```

### 5.1 tmp与raw的区别

| 目录 | 性质 | 是否提炼 | 用途 |
|------|------|:--------:|------|
| `raw/tmp/` | 暂存区 | ❌ | 零散笔记、待整理内容 |
| `raw/其他文件夹/` | 正式区 | ✅ | 主题明确、可提炼的笔记 |

### 5.2 入库即提炼的优势

- **节省token**：避免后续重复读取raw文件
- **提高效率**：一次操作完成入库+提炼
- **流程简化**：减少操作步骤和日志记录

## 六、图片处理规范

### 6.1 处理原则
- **所有图片必须base64嵌入**：不保留外部图片文件引用
- **处理时机**：在笔记命名并放入raw目录后、提炼前完成
- **保持原图质量**：base64编码不损失图片质量

### 6.2 支持的图片格式
- PNG（推荐，适合截图、图表）
- JPG/JPEG（适合照片）
- GIF（适合动图）
- SVG（适合矢量图）
- WebP（现代格式）

### 6.3 处理流程

```
检查笔记中的图片引用
    │
    ▼
┌─────────────────────────────────────────┐
│ 1. 识别图片引用格式                       │
│    - Obsidian格式：![[图片名.png]]        │
│    - Markdown格式：![alt](路径/图片.png)  │
│    - HTML格式：<img src="路径/图片.png">  │
└────────┬────────────────────────────────┘
         │
         ▼
┌─────────────────────────────────────────┐
│ 2. 定位图片文件                           │
│    - 同目录下的图片文件                    │
│    - 附件文件夹中的图片                    │
│    - 绝对路径引用的图片                    │
└────────┬────────────────────────────────┘
         │
         ▼
┌─────────────────────────────────────────┐
│ 3. 转换为base64编码                       │
│    - 读取图片二进制数据                    │
│    - 转换为base64字符串                    │
│    - 生成data URI格式                     │
└────────┬────────────────────────────────┘
         │
         ▼
┌─────────────────────────────────────────┐
│ 4. 替换图片引用                           │
│    - 替换为：![](data:image/类型;base64,) │
│    - 保持alt文本（如有）                   │
└────────┬────────────────────────────────┘
         │
         ▼
┌─────────────────────────────────────────┐
│ 5. 清理原始图片文件                       │
│    - 删除已嵌入的图片文件                  │
│    - 清理空的附件文件夹（如适用）           │
└─────────────────────────────────────────┘
```

### 6.4 base64嵌入格式

#### 标准格式
```markdown
![图片描述](data:image/png;base64,iVBORw0KGgoAAAANSUhEUgAA...)
```

#### Obsidian格式转换
```markdown
# 原始格式
![[Pasted image 20260604184631.png]]

# 转换后格式
![](data:image/png;base64,iVBORw0KGgoAAAANSUhEUgAA...)
```

### 6.5 大小限制建议
- **单张图片**：建议不超过 5MB
- **笔记总大小**：建议不超过 10MB（含所有base64图片）
- **超大图片**：考虑压缩或降低分辨率

### 6.6 特殊情况处理

#### 图片文件不存在
- 记录日志标记 `REVIEW`
- 保留原始引用，不进行转换
- 在日志中备注"图片文件缺失"

#### 网络图片引用
- 不下载网络图片
- 保留原始URL引用
- 在日志中标记"网络图片，未嵌入"

#### 多图片批量处理
```python
# Python示例代码
import base64
import re

def embed_images(content, base_path):
    # 匹配Obsidian格式
    pattern = r'!\[\[(.+?\.(?:png|jpg|jpeg|gif|svg|webp))\]\]'
    
    def replace_match(match):
        filename = match.group(1)
        filepath = os.path.join(base_path, filename)
        
        if os.path.exists(filepath):
            with open(filepath, 'rb') as f:
                data = base64.b64encode(f.read()).decode('utf-8')
            ext = filename.split('.')[-1].lower()
            return f'![](data:image/{ext};base64,{data})'
        return match.group(0)
    
    return re.sub(pattern, replace_match, content, flags=re.IGNORECASE)
```

## 七、日志记录

遵循 `log_standard.md` 规范。

### 7.1 操作类型
- 放入`tmp/`：只记录 `RAW_IN`
- 放入`raw/`其他文件夹：同时记录 `RAW_IN` 和 `EXTRACT`

### 7.2 日志示例
```
# 放入tmp（不提炼）
15:30 RAW_IN OK raw/tmp/临时笔记.md

# 放入raw（同时提炼）
15:35 RAW_IN OK raw/嵌入式开发/I2C通信协议/01_基础.md
15:35 EXTRACT OK core/I2C通信协议.md
```

## 八、禁止事项

- ❌ 修改原文观点、数据、表述
- ❌ 保留外部图片引用（必须转为base64嵌入）
- ❌ 跳过有效性检查直接入库
- ❌ 文件夹嵌套超过 5 层
- ❌ 在文件名中使用非法字符 `/\:*?"<>|`
- ❌ 将不同主题的笔记混放在同一叶子文件夹
