# 核心提炼规范

## 目标
将 `raw/` 下同一主题的所有原始笔记，整合为一份**清晰易读、结构良好**的核心文档，存入 `core/`，方便快速复习和实践中查阅。

## 核心原则
1. **清晰优先，而非精简优先**：不追求极致的文字压缩，追求的是表达清晰、一目了然。
2. **结构化表达**：内容多的时候，用列表、表格、层级来组织，避免大段文字墙。
3. **去论证，留知识**：删除推导过程、实验细节、背景故事，但保留知识本身，允许为讲清楚一个概念而写足够的解释。
4. **可溯源**：每条知识模块必须附带原文跳转链接，看不懂时一键回去看完整论证。

## 一、提炼标准

### 1.1 应该保留的内容
- **核心概念**：这个东西是什么，用清晰的语言讲明白（不追求一句话）
- **关键原理**：运作机制的本质结论（不需要推导，但需要讲清楚）
- **使用要点**：实践中需要注意什么、常见陷阱、适用场景、最佳实践
- **关键数据/参数**：重要的数字、阈值、配置建议
- **公式**：保留公式本身和每个符号的含义，不保留推导
- **好的例子**：如果原文的例子能帮助快速理解，可以保留（精简后）

### 1.2 应该删除的内容
- 推导步骤、数学证明过程
- 实验设计、数据集描述、调参过程记录
- 背景故事、作者个人经历、心路历程
- 与主题无关的岔开话题
- 明显的废话、重复论述、客套话

### 1.3 表述风格
- **易读第一**：用平实的语言解释，不必像定理一样精炼
- **允许必要长度**：一个概念如果需要 200 字才能讲清楚，就写 200 字
- **禁止大段文字墙**：超过 5 行的纯文字段落应考虑拆分为列表或子条目
- **善用格式**：
  - 并列概念 → 无序列表
  - 步骤/流程 → 有序列表
  - 对比/对照 → 表格
  - 层级关系 → 嵌套列表或标题
  - 流程/关系 → 可描述用 ASCII 图或 mermaid

### 1.4 表述风格对比
同一段原文的两种提炼方式：

> ❌ 过度精简（不推荐）：
> - 正则化：对模型复杂度施加惩罚，防过拟合。L1 产生稀疏权重，L2 缩小权重。

> ✅ 清晰表达（推荐）：
> ### 正则化
> 一种防止模型过拟合的技术，在损失函数中增加一个惩罚项，限制模型复杂度。
> 
> 常见两种：
> - **L1 正则化**：惩罚权重的绝对值之和
>   - 效果：让部分权重精确变为 0，产生稀疏模型
>   - 适用：需要特征选择的场景
> - **L2 正则化**：惩罚权重的平方和
>   - 效果：让所有权重整体缩小，但不为 0
>   - 适用：大多数场景的默认选择，防止权重过大
> 
> | 对比 | L1 | L2 |
> |------|----|----|
> | 惩罚项 | 权重绝对值之和 | 权重平方和 |
> | 结果 | 稀疏（部分为0） | 平滑（整体缩小） |
> | 适用 | 特征选择 | 通用防过拟合 |
>

## 二、文件组织规范

### 2.1 文件命名
- 格式：`主题名.md`
- 直接使用 `raw/` 中对应的主题文件夹名称
- 示例：
  ```
  raw/计算机科学/机器学习/    →    core/机器学习.md
  raw/个人成长/时间管理/      →    core/时间管理.md
  ```

### 2.2 一个主题一个文件
- 一个 `core/xxx.md` 文件聚合该主题下所有 `raw/` 笔记的提炼内容
- 不按单篇笔记拆分文件，而是按知识领域聚合

### 2.3 主题对应关系
- `core/` 的主题文件应与 `raw/` 的某一层级主题文件夹对应
- 建议以 `raw/` 的第 3~4 层主题为提炼单位
- 示例：
  ```
  raw/计算机科学/人工智能/机器学习/   →   core/机器学习.md
  raw/个人成长/思维模型/             →   core/思维模型.md
  ```

## 三、文档内部结构规范

### 3.1 分层标题
使用 Markdown 标题层级组织知识，形成树状结构：
```
# 主题名（一级标题）

## 子领域A（二级标题）
### 知识点1（三级标题）
### 知识点2

## 子领域B
### 知识点3
### 知识点4
```

### 3.2 分层原则
- **二级标题（##）**：该主题下的子领域划分
- **三级标题（###）**：具体知识点条目
- **四级标题（####）**：知识点的细分（仅在内容足够多时使用）
- 最多使用到 **四级标题**

### 3.3 知识点条目格式模板

```markdown
### 知识点名称

清晰的概念解释，允许必要的长度，以讲明白为准。

- **要点1**：说明
- **要点2**：说明
  - 子要点（如需要）
- **注意**：常见的坑或关键提示

📄 [原文]( 相对路径 )
```

### 3.4 信息组织辅助手段
当内容较多时，积极使用以下方式增强可读性：

**表格**（用于对比/对照）：
```markdown
| 方法 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| 方法A | ... | ... | ... |
| 方法B | ... | ... | ... |
```

**嵌套列表**（用于层级关系）：
```markdown
- 大类1
  - 子类1-1
    - 细节A
    - 细节B
  - 子类1-2
- 大类2
```

**流程图/示意图（ASCII 或 mermaid）**：
按需使用，不强制。

### 3.5 链接格式规范
- 使用**相对路径**，确保知识库整体移动后仍可跳转
- 链接文本使用 `[原文XX]` 格式，XX为对应raw文件的序号（如 `原文01`、`原文02`）
- 链接放在每个知识模块的末尾
- 如果一个知识点来源于多篇笔记：
  ```markdown
  📄 [原文01](../raw/xxx/01_xxx.md) · [原文02](../raw/xxx/02_xxx.md)
  ```

### 3.6 章节顺序映射规范
- core笔记的**章节顺序**应与raw文件的**序号顺序**一致
- 二级标题（##）作为主要章节，对应raw中的文件顺序
- 例如：raw中有01_基础、02_实现、03_测试，那么core中就有：
  ```markdown
  ## 一、基础概念（对应01）
  ## 二、实现方法（对应02）
  ## 三、测试验证（对应03）
  ```
- 这样读者按章节顺序阅读 = 按raw文件顺序学习

## 五、质量检查清单

提炼完成后，逐条检查：

- [ ] 表述是否清晰，能否轻松理解？
- [ ] 是否删除了推导过程和不必要的背景？
- [ ] 大段文字是否已拆分为列表、表格等易读形式？
- [ ] 对比性内容是否使用了表格？
- [ ] 分层结构是否合理，同类知识是否归在一起？
- [ ] 原文链接是否正确、可点击、使用相对路径？
- [ ] 标题层级是否控制在四级以内？

## 六、增量更新规范

### 6.1 来源清单
每个core文件开头必须包含**来源清单**，记录已提炼的raw文件：

```markdown
> **来源清单**（已提炼）：
> - [x] 01_文件名A
> - [x] 02_文件名B
> - [ ] 03_文件名C（如有未提炼的）
>
> **更新时间**：YYYY-MM-DD
```

### 6.2 索引状态标记
在 `index/全文目录.md` 中，每个raw文件需标记提炼状态：
- `✅ 已提炼`：内容已整合到core
- `⬜ 待提炼`：内容尚未提炼

### 6.3 增量更新触发条件
以下情况需要更新core笔记：
- raw中**新增**了笔记 → 在core中添加对应知识点，更新来源清单
- raw中**修改**了笔记 → 检查core是否需要同步更新
- raw中**删除**了笔记 → 从来源清单中移除，检查core中对应内容是否保留

### 6.4 增量更新流程
```
发现raw有变动
    ↓
检查index/全文目录.md → 确认提炼状态
    ↓
如果是新增/修改 → 提炼到core → 更新来源清单
    ↓
更新index/全文目录.md → 标记"✅ 已提炼"
    ↓
记录日志
```

## 八、禁止事项

- ❌ 为了精简而牺牲清晰度
- ❌ 保留推导过程和实验细节
- ❌ 大段纯文字不进行结构优化
- ❌ 忘记添加原文链接
- ❌ 使用绝对路径做链接
- ❌ 把不同主题的提炼内容混在一个文件里
