## 一、I2C 协议基础

### 1.1 基本概念

| 项目 | 内容 |
|------|------|
| 全称 | Inter-Integrated Circuit（内部集成电路） |
| 正确写法 | **I²C**（I 的平方 C） |
| 开发者 | 飞利浦公司（20 世纪 80 年代初期） |
| 协议性质 | 开放通用协议（知识产权已公开） |
| 定位 | 系统内多个 IC 之间的**低速串行通讯** |
| 特点 | 引脚少、硬件简单、可扩展性强、通讯稳定、成本低 |

📄 [原文01: I2C协议基础与AT24C02详解](../raw/嵌入式开发/I2C通信协议/01_I2C协议基础与AT24C02详解.md)

### 1.2 与串口对比

| 对比项 | UART（串口） | I²C |
|--------|-------------|-----|
| 信号线 | TX + RX（两根数据线） | SCL + SDA（一根时钟线 + 一根数据线） |
| 同步方式 | **异步**（无时钟线） | **同步**（有时钟线） |
| 双工模式 | **全双工**（收发各一根线） | **半双工**（一根数据线分时复用） |
| 连接方式 | **点对点**（TX↔RX 交叉连接） | **总线**（所有设备并联） |
| 输出模式 | **推挽输出** | **开漏输出 + 上拉电阻** |
| 传输顺序 | **低位先行** (LSB First) | **高位先行** (MSB First) |
| 应答机制 | **无** | **有**（ACK/NACK） |
| 寻址方式 | 无（点对点，不需要寻址） | **设备地址**（7 位或 10 位） |

📄 [原文01: I2C协议基础与AT24C02详解](../raw/嵌入式开发/I2C通信协议/01_I2C协议基础与AT24C02详解.md)

### 1.3 传输速率

| 模式 | 速率 | 说明 |
|------|------|------|
| 标准模式 | **100 kbps** | 与串口 115200 波特率同量级 |
| 快速模式 | **400 kbps** | 大多数设备支持 |
| 高速模式 | **3.4 Mbps** | 仅部分设备支持 |

## 二、物理层 — 硬件连接

### 2.1 两根信号线

| 线 | 全称 | 功能 | 驱动方 |
|----|------|------|--------|
| **SCL** | Serial Clock | 串行时钟线，由主设备控制，用于数据收发同步 | **主设备** |
| **SDA** | Serial Data | 串行数据线，双向传输，半双工 | 主设备或从设备（分时复用） |

📄 [原文01: I2C协议基础与AT24C02详解](../raw/嵌入式开发/I2C通信协议/01_I2C协议基础与AT24C02详解.md)

### 2.2 总线拓扑

```
         VCC
          │
         [R] 上拉电阻
          │
SCL ──────┼────────────── 所有设备 SCL 引脚并联
          │
SDA ──────┼────────────── 所有设备 SDA 引脚并联
          │
   ┌──────┼──────┬──────────┐
   │      │      │          │
 主设备  从设备1  从设备2  从设备N
```

- 所有设备**并联**到同一条总线
- 总线空闲时，SDA 和 SCL 通过**上拉电阻**保持高电平
- 与串口的点对点连接不同，I²C 是**一主多从**或多主多从的总线结构

📄 [原文01: I2C协议基础与AT24C02详解](../raw/嵌入式开发/I2C通信协议/01_I2C协议基础与AT24C02详解.md)

### 2.3 开漏输出与线与特性

| 项目 | 说明 |
|------|------|
| 输出模式 | 所有设备采用**开漏输出** |
| 默认状态 | 高阻态，由上拉电阻拉至高电平（逻辑 1） |
| 占用总线 | 设备输出低电平（接地），总线被拉低（逻辑 0） |
| 线与特性 | 任何一个设备输出 0，总线即为 0；所有设备输出高阻态，总线才为 1 |

> 开漏输出 + 上拉电阻实现了"线与"逻辑，是 I²C 总线多设备共存的基础。

📄 [原文01: I2C协议基础与AT24C02详解](../raw/嵌入式开发/I2C通信协议/01_I2C协议基础与AT24C02详解.md)

### 2.4 主设备与从设备

| 角色 | 英文 | 特征 |
|------|------|------|
| **主设备** | Master | 控制 SCL 时钟信号，发起通讯，发送设备地址 |
| **从设备** | Slave | 接收时钟信号，响应主设备请求，拥有唯一地址 |

- **SCL 时钟线**：正常情况下仅由主设备驱动
- **特殊情况**：从设备可通过拉低 SCL 请求**时钟扩展**（Clock Stretching），通知主设备放慢速度
- **SDA 数据线**：主设备和从设备都可以驱动（半双工，分时复用）

## 三、协议层 — 数据传输规则

### 3.1 传输顺序

| 项目 | 说明 |
|------|------|
| 传输方向 | **高位先行（MSB First）** |
| 与串口对比 | 串口为低位先行（LSB First），I²C 相反 |

📄 [原文01: I2C协议基础与AT24C02详解](../raw/嵌入式开发/I2C通信协议/01_I2C协议基础与AT24C02详解.md)

### 3.2 起始信号（S）与停止信号（P）

**前提条件：** SCL 为**高电平**时，SDA 发生跳变

| 信号 | SCL | SDA 变化 | 简写 | 含义 | 总线状态 |
|------|:---:|----------|:----:|------|----------|
| **起始信号** | **高电平** | 高 → 低（**下降沿**） | **S** (Start) | 主设备占用总线，开始通讯 | 进入通讯 |
| **停止信号** | **高电平** | 低 → 高（**上升沿**） | **P** (Stop) | 主设备释放总线，结束通讯 | 恢复空闲 |

**直观记忆：**
```
空闲（高）→ 起始（拉低）→ 数据传输（高低交替）→ 停止（拉高）→ 空闲（高）
  高           低              锯齿波               高             高

类比：跳进坑（S）→ 在坑里干活（数据传输）→ 爬出坑（P）
```

📄 [原文01: I2C协议基础与AT24C02详解](../raw/嵌入式开发/I2C通信协议/01_I2C协议基础与AT24C02详解.md)

### 3.3 数据有效性

| SCL 电平 | SDA 状态 | 含义 |
|----------|----------|------|
| **高电平** | **保持稳定**（高或低） | 数据采样时刻，读取 SDA 的 0 或 1 |
| **高电平** | **发生跳变** | 起始信号（高→低）或停止信号（低→高） |
| **低电平** | **可以改变** | 数据翻转时刻，为下一次采样准备新的电平 |

**核心规则：**
- SCL 高电平期间，SDA **必须保持稳定**（数据有效）
- SCL 低电平期间，SDA **可以改变**（准备下一位数据）

**传输一位数据的完整过程：**
```
① SCL 低电平期间 → SDA 设置好目标电平（高或低）
② SCL 低→高       → SDA 保持稳定，从设备采样
③ SCL 高→低       → 采样结束
④ SCL 低电平期间 → SDA 翻转为下一位数据的电平
```

> **关键：** SCL 高电平时发生 SDA 跳变 = 起始/停止信号；保持稳定 = 数据采样。

📄 [原文01: I2C协议基础与AT24C02详解](../raw/嵌入式开发/I2C通信协议/01_I2C协议基础与AT24C02详解.md)

### 3.4 数据帧结构

一次完整的 I²C 通讯包含以下部分：

```
┌─────┬──────────────┬─────────────┬─────────┬─────┐
│  S  │ 设备地址(7位) │ R/W 方向位  │ 数据(8位)│ ACK │  → 可多次传输数据
└─────┴──────────────┴─────────────┴─────────┴─────┘
起始   从机地址        读/写控制     有效数据   应答
```

📄 [原文01: I2C协议基础与AT24C02详解](../raw/嵌入式开发/I2C通信协议/01_I2C协议基础与AT24C02详解.md)

### 3.5 设备地址

#### 地址格式

```
┌────────────────────────┬─────────┐
│  设备地址（7位或10位）    │ R/W 位   │
└────────────────────────┴─────────┘
```

#### 地址位数

| 地址位数 | 可寻址设备数 | 说明 |
|----------|------------|------|
| **7 位** | 2^7 = **128** | 一般场景够用，最常用 |
| **10 位** | 2^10 = **1024** | 扩展模式，设备数量更多时使用 |

#### 读写方向位（R/W）

| 位值 | 含义 | 说明 |
|:----:|------|------|
| **0** | **写** | 主设备向从设备写入数据 |
| **1** | **读** | 主设备从从设备读取数据 |

> R/W 位是从**主设备的视角**定义的。

📄 [原文01: I2C协议基础与AT24C02详解](../raw/嵌入式开发/I2C通信协议/01_I2C协议基础与AT24C02详解.md)

## 四、应答机制（ACK / NACK）

### 4.1 为什么需要应答

- SDA 只有一根线，收发共用
- 发送方必须确认接收方是否正确收到数据
- 没有应答机制，后续数据传输可能混乱

📄 [原文01: I2C协议基础与AT24C02详解](../raw/嵌入式开发/I2C通信协议/01_I2C协议基础与AT24C02详解.md)

### 4.2 应答时序

```
发送方                              接收方
  │                                  │
  │  传输 8 位数据（一个字节）          │
  │ ───────────────────────────────→ │
  │                                  │
  │  释放 SDA（输出高/高阻态）          │
  │                                  │
  │  ←───────────────────────────── │  接收方拉低 SDA = ACK
  │     （第 9 个时钟周期）             │
  │                                  │
```

### 4.3 应答类型

| 类型 | 英文 | SDA 电平 | 含义 |
|------|------|:--------:|------|
| **应答** | ACK (Acknowledge) | **低电平 (0)** | 接收成功，可继续传输 |
| **非应答** | NACK (Not Acknowledge) | **高电平 (1)** | 接收失败或无应答，需停止或重试 |

### 4.4 应答过程详解

```
第 1~8 个时钟周期：发送方驱动 SDA，传输 8 位数据
    ↓
发送方释放 SDA（输出高阻态，SDA 被上拉至高电平）
    ↓
第 9 个时钟周期：接收方控制 SDA
    ├── 拉低 SDA → ACK（收到数据，应答）
    └── 保持高电平 → NACK（未收到或不愿接收）
    ↓
接收方释放 SDA，准备下一轮传输
```

**NACK 的作用：**
- 从设备收到 NACK 后知道主机不需要更多数据
- 数据总线恢复空闲
- 主机可以发送停止信号（P）结束通讯

📄 [原文01: I2C协议基础与AT24C02详解](../raw/嵌入式开发/I2C通信协议/01_I2C协议基础与AT24C02详解.md)

## 五、I²C 通讯完整流程

```
① 主设备发送起始信号（S）
        ↓
② 主设备发送 7 位从设备地址 + 1 位 R/W
        ↓
③ 从设备应答（ACK/NACK）
        ↓
④ 数据传输（每 8 位数据 + 1 位应答，可多次）
   ├── 写操作：主设备发数据 → 从设备应答
   └── 读操作：从设备发数据 → 主设备应答
        ↓
⑤ 主设备发送停止信号（P）
```

## 六、AT24C02 EEPROM 芯片详解

### 6.1 基本参数

| 参数 | 数值 |
|------|------|
| 全称 | Electrically Erasable Programmable Read-Only Memory |
| 容量 | **256 字节**（2 Kbit） |
| 页大小 | **16 字节** |
| 页数 | 16 页 |
| 写入周期 | **5 ms** |
| I²C 模式 | 标准模式 100kHz / 快速模式 400kHz |
| 供电电压 | 1.7V ~ 5.5V |
| 设备类型 ID | 1010（高 4 位固定） |

📄 [原文01: I2C协议基础与AT24C02详解](../raw/嵌入式开发/I2C通信协议/01_I2C协议基础与AT24C02详解.md)

### 6.2 内部地址结构

```
8 位地址（0x00 ~ 0xFF）表示 256 个字节

地址位划分：
┌──────────────┬──────────────┐
│  高 4 位       │  低 4 位       │
│  页地址        │  页内地址       │
│  (0~15)       │  (0~15)       │
└──────────────┴──────────────┘

例：地址 0x3A → 页地址 = 0x3，页内地址 = 0xA
```

| 页号 | 地址范围 |
|:----:|----------|
| 0 | 0x00 ~ 0x0F |
| 1 | 0x10 ~ 0x1F |
| ... | ... |
| 15 | 0xF0 ~ 0xFF |

📄 [原文01: I2C协议基础与AT24C02详解](../raw/嵌入式开发/I2C通信协议/01_I2C协议基础与AT24C02详解.md)

### 6.3 设备地址 vs 存储内部地址

这是两个**完全不同**的概念，必须严格区分：

| 对比项 | 设备地址 | 存储内部地址 |
|--------|----------|------------|
| 含义 | 识别总线上的**哪个设备** | 设备内部的**哪个存储单元** |
| 位数 | 7 位（+ 1 位读写方向位） | 8 位 |
| 范围 | 最多 128 个设备 | 0x00 ~ 0xFF（256 个字节） |
| 发送时机 | 通讯开始后**第一个**发送 | 设备地址之后发送 |
| 用途 | 主设备在总线上寻址从设备 | 指定从设备内部的读写位置 |

📄 [原文01: I2C协议基础与AT24C02详解](../raw/嵌入式开发/I2C通信协议/01_I2C协议基础与AT24C02详解.md)

### 6.4 AT24C02 设备地址

#### 地址构成

```
7 位设备地址 = [1 0 1 0] [A2] [A1] [A0]
                 固定4位    ← 3位由引脚决定 →

本实验：A2 = A1 = A0 = 0（全接 GND）
设备地址 = 0b1010000 = 0x50
```

#### 设备码（含读写方向位）

```
完整 1 字节 = [设备地址 7 位] [R/W 1 位]

写操作：0b1010000 + 0 = 0b10100000 = 0xA0
读操作：0b1010000 + 1 = 0b10100001 = 0xA1
```

#### 代码中的宏定义

```c
#define AT24C02_WRITE_ADDR   0xA0    // 写地址
#define AT24C02_READ_ADDR    0xA1    // 读地址
```

📄 [原文01: I2C协议基础与AT24C02详解](../raw/嵌入式开发/I2C通信协议/01_I2C协议基础与AT24C02详解.md)

### 6.5 引脚功能速查

| 引脚 | 名称 | 连接 | 功能 |
|:----:|------|------|------|
| 1 | A0 | GND | 设备地址 bit0 = 0 |
| 2 | A1 | GND | 设备地址 bit1 = 0 |
| 3 | A2 | GND | 设备地址 bit2 = 0 |
| 4 | GND | GND | 电源地 |
| 5 | SDA | PB11 | I²C 数据线（经上拉电阻接 3.3V） |
| 6 | SCL | PB10 | I²C 时钟线（经上拉电阻接 3.3V） |
| 7 | WP | GND | 写保护关闭（低电平允许读写） |
| 8 | VCC | 3.3V | 电源 |

📄 [原文01: I2C协议基础与AT24C02详解](../raw/嵌入式开发/I2C通信协议/01_I2C协议基础与AT24C02详解.md)

### 6.6 两个地址的发送顺序

```
主机发送顺序：

① 起始信号(S)
② 设备地址 + R/W   ← 第一个字节：告诉总线上哪个设备、读还是写
③ 存储内部地址      ← 第二个字节：告诉该设备从哪个存储单元操作
④ 数据（写入）或 接收数据（读取）
⑤ 停止信号(P)
```

```
示例：向 AT24C02 的地址 0x3A 写入数据 0x55

[S] [0xA0] [ACK] [0x3A] [ACK] [0x55] [ACK] [P]
      ↑               ↑               ↑
   设备码(写)      存储内部地址       写入数据
```

## 七、AT24C02 读写时序

### 7.1 写入一个字节（Byte Write）

#### 完整时序

```
[S] → [0xA0] → ACK → [内部地址] → ACK → [数据] → ACK → [P]
                                   等待 5ms 写入周期
```

#### 步骤详解

| 步骤 | 方向 | 内容 | 应答 |
|:----:|------|------|:----:|
| 1 | 主→从 | 起始信号 S | — |
| 2 | 主→从 | 设备码 **0xA0**（7位地址1010000 + 写0） | ACK |
| 3 | 主→从 | **内部存储地址**（0x00~0xFF，1字节） | ACK |
| 4 | 主→从 | **写入数据**（1字节） | ACK |
| 5 | 主→从 | 停止信号 P | — |
| 6 | — | **等待 5ms**（写入周期） | — |

📄 [原文01: I2C协议基础与AT24C02详解](../raw/嵌入式开发/I2C通信协议/01_I2C协议基础与AT24C02详解.md)

### 7.2 读取一个字节 — 两种方式

#### 当前地址读取（Current Address Read）

```
[S] → [设备地址码 + R(1)] → 从设备 ACK → [从设备发送数据] → 主设备 NACK → [P]
```

**读取位置：** 从 AT24C02 内部地址计数器（Address Counter）当前指向的位置开始。

**特点：** 只能从上次读写位置**顺序继续**读取，不能指定任意地址。

📄 [原文01: I2C协议基础与AT24C02详解](../raw/嵌入式开发/I2C通信协议/01_I2C协议基础与AT24C02详解.md)

#### 随机地址读取（Random Address Read）— 假写真读

**为什么需要假写真读：**

| 直接读的问题 | 解决方案 |
|-------------|----------|
| 给读地址(R=1)后，总线控制权立即交给从设备 | 先用**写地址**(R=0)保持主设备控制权 |
| 无法在读取前指定内部地址 | 用"假写"将目标地址写入地址计数器 |

**假写阶段（设置地址计数器）：**
```
[S] → [设备地址码 + W(0)] → ACK → [内部地址] → ACK
                                        ↑
                              写入地址计数器（非存储单元）
```

**真读阶段（从指定地址读取数据）：**
```
[Sr] → [设备地址码 + R(1)] → ACK → [从设备发送数据] → 主设备 NACK → [P]
```

**完整流程图：**
```
┌──────────── 假写（主设备控制总线）────────────┐
│                                               │
│  [S] → [0xA0(W)] → ACK → [0x3A] → ACK       │
│   ↑       ↑                   ↑               │
│  起始   写地址             写入地址计数器       │
│                                               │
└───────────────────────────────────────────────┘
              ↓ 总线控制权不变
┌──────────── 真读（切换为从设备控制）──────────┐
│                                               │
│  [Sr] → [0xA1(R)] → ACK → [数据] → NACK → [P]│
│   ↑       ↑                   ↑       ↑       │
│  重复起始 读地址          从设备发送  够了别发了 │
│                    总线控制权交给从设备         │
└───────────────────────────────────────────────┘
```

#### 两种读取方式对比

| 对比项 | 当前地址读取 | 随机地址读取（假写真读） |
|--------|:----------:|:---------------------:|
| 起始步骤 | 直接发读地址 | 先假写（写地址+内部地址） |
| 起始信号数 | 1 个 S | 2 个（S + Sr） |
| 内部地址指定 | 不可（用计数器当前位置） | 可指定任意地址 |
| 地址计数器 | 使用当前值 | 被更新为指定值 |
| 读取位置 | 上次操作的下一位置 | 主设备指定的任意位置 |
| 适用场景 | 顺序连续读取 | **随机访问**任意存储单元 |

📄 [原文01: I2C协议基础与AT24C02详解](../raw/嵌入式开发/I2C通信协议/01_I2C协议基础与AT24C02详解.md)

### 7.3 连续写入多个字节（Page Write）

#### 时序

```
[S] → [0xA0] → [ACK] → [内部地址] → [ACK]
  → [数据1] → [ACK] → [数据2] → [ACK] → ... → [数据N] → [ACK]
  → [P]
```

#### 页写入限制

| 规则 | 说明 |
|------|------|
| 每次最多 **16 字节** | AT24C02 页大小 = 16 字节 |
| **不可跨页** | 写到页末后地址自动**绕回**页首 |
| 绕回后果 | 页首数据被**覆盖**，相当于数据丢失 |
| 解决方案 | 分多次写入，每次不超 16 字节且不跨页 |

```
页内地址循环：
0x00 → 0x01 → ... → 0x0F → 0x00（绕回覆盖）→ 0x01 → ...
```

📄 [原文01: I2C协议基础与AT24C02详解](../raw/嵌入式开发/I2C通信协议/01_I2C协议基础与AT24C02详解.md)

### 7.4 连续读取多个字节（Sequential Read）

#### 时序（随机地址连续读）

```
假写：[S] → [0xA0] → [ACK] → [起始内部地址] → [ACK]

真读：[Sr] → [0xA1] → [ACK]
  → [数据1] ← 主机 ACK（还要）
  → [数据2] ← 主机 ACK（还要）
  → ...
  → [数据N] ← 主机 NACK（够了）
  → [P]
```

#### ACK 与 NACK 的区别

| 主机应答 | 含义 | 从设备行为 |
|:--------:|------|-----------|
| **ACK** | 还要继续读 | 地址计数器 +1，发送下一字节 |
| **NACK** | 读取结束 | 停止发送，释放数据总线 |

📄 [原文01: I2C协议基础与AT24C02详解](../raw/嵌入式开发/I2C通信协议/01_I2C协议基础与AT24C02详解.md)

### 7.5 AT24C02 读取模式分类

| 模式 | 英文 | 操作 | 地址来源 |
|------|------|------|----------|
| **当前地址读** | Current Address Read | 直接读 | 地址计数器当前位置 |
| **随机地址读** | Random Address Read | 假写真读 | 主机指定地址 |
| **当前地址连续读** | Sequential Read (Current) | 当前地址读 + 多次 ACK | 从计数器位置开始 |
| **随机地址连续读** | Sequential Read (Random) | 假写真读 + 多次 ACK | 从主机指定地址开始 |

## 八、地址计数器（Address Counter）

```
AT24C02 内部维护一个 8 位地址计数器：

写入操作时：
  → 计数器被设置为写入的内部地址
  → 写入完成后计数器 +1

读取操作时：
  → 从计数器指向的地址开始读取
  → 读取完成后计数器 +1

假写操作时：
  → 计数器被设置为假写指定的地址
  → 不执行真正的存储写入
```

**作用：**
- 当前地址读取时，从计数器指向的位置开始
- 连续读取时，自动递增访问下一个存储单元
- 假写操作实质是设置计数器的值

## 十、I²C 协议层函数实现

### 10.1 宏定义

#### ACK / NACK 常量

```c
#define I2C_ACK    0    // 应答：低电平
#define I2C_NACK   1    // 非应答：高电平
```

| 常量 | 值 | 对应 SDA 电平 | 含义 |
|------|:--:|:------------:|------|
| `I2C_ACK` | 0 | 低电平 | 接收成功，应答 |
| `I2C_NACK` | 1 | 高电平 | 接收失败 / 不再接收 |

#### SCL 控制宏

```c
#define SCL_HIGH()    (GPIOB->ODR |=  (1 << 10))   // PB10 拉高
#define SCL_LOW()     (GPIOB->ODR &= ~(1 << 10))   // PB10 拉低
```

#### SDA 写入控制宏

```c
#define SDA_HIGH()    (GPIOB->ODR |=  (1 << 11))   // PB11 拉高
#define SDA_LOW()     (GPIOB->ODR &= ~(1 << 11))   // PB11 拉低
```

#### SDA 读取宏

```c
#define READ_SDA()    (GPIOB->IDR & (1 << 11))     // 读取 PB11 电平
```

> **关键：** GPIO 配置为输出模式时，IDR 仍可读取引脚实际电平，因此无需切换输入/输出模式即可读取 SDA。

#### 延迟宏

```c
#define I2C_DELAY()   Delay_us(10)
```

| 宏 | 延迟 | 计算依据 |
|----|------|----------|
| `I2C_DELAY()` | 10 μs | I²C 标准模式 100kHz → 周期 10μs |

### 10.3 I2C_Init() — GPIO 初始化

#### 配置要求

| 引脚 | 功能 | GPIO 模式 | CNF | MODE |
|------|------|-----------|:---:|:----:|
| **PB10** | SCL 时钟线 | 通用开漏输出，50MHz | 01 | 11 |
| **PB11** | SDA 数据线 | 通用开漏输出，50MHz | 01 | 11 |

> CRH 寄存器（引脚 8~15），每引脚占 4 位：`[CNF1:CNF0][MODE1:MODE0]`

#### 代码实现

```c
void I2C_Init(void)
{
    /* 1. 开启 GPIOB 时钟 */
    RCC->APB2ENR |= RCC_APB2ENR_IOPBEN;

    /* 2. 配置 PB10 和 PB11 为通用开漏输出，50MHz */
    // MODE[1:0] = 11（50MHz 输出）
    GPIOB->CRH |=  (GPIO_CRH_MODE10 | GPIO_CRH_MODE11);
    
    // CNF[1:0] = 01（通用开漏输出）
    GPIOB->CRH &= ~(GPIO_CRH_CNF10 | GPIO_CRH_CNF11);   // 先清零
    GPIOB->CRH |=  (GPIO_CRH_CNF10_0 | GPIO_CRH_CNF11_0); // 再置 01
}
```

#### 为什么用开漏输出

| 输出值 | NMOS 状态 | 引脚状态 | 总线状态 |
|:------:|----------|----------|----------|
| 0 | 导通 | 接地 | **低电平**（占用总线） |
| 1 | 关断 | 高阻态 | **高电平**（上拉电阻拉高，释放总线） |

> 开漏输出 + 外部上拉电阻完美匹配 I²C 总线的线与逻辑。

### 10.4 I2C_Start() — 发送起始信号

#### 时序回顾

```
起始信号：SCL 高电平期间，SDA 产生下降沿（高→低）

SCL:  ──────────┐           ┌──────────
                 └───────────┘
SDA:  ──┐                   （数据传输...）
        └───────────────────
         ↑ 起始信号
```

#### 代码实现

```c
void I2C_Start(void)
{
    /* ① 两线拉高（空闲状态） */
    SDA_HIGH();
    SCL_HIGH();
    I2C_DELAY();

    /* ② SCL 保持高，SDA 拉低（下降沿 = 起始信号） */
    SDA_LOW();
    I2C_DELAY();
}
```

### 10.5 I2C_Stop() — 发送停止信号

#### 时序回顾

```
停止信号：SCL 高电平期间，SDA 产生上升沿（低→高）

SCL:  ──────────┐           ┌──────────
                 └───────────┘
SDA:  ────────────────────┐
                          └──────────
                           ↑ 停止信号
```

#### 代码实现

```c
void I2C_Stop(void)
{
    /* ① SCL 拉高，SDA 拉低（准备状态） */
    SDA_LOW();
    SCL_HIGH();
    I2C_DELAY();

    /* ② SCL 保持高，SDA 拉高（上升沿 = 停止信号） */
    SDA_HIGH();
    I2C_DELAY();
}
```

#### Start 与 Stop 对比

| 操作 | 初始 SDA | 最终 SDA | SCL |
|------|:--------:|:--------:|:---:|
| **Start** | 高（拉高） | **低**（拉低） | 保持高 |
| **Stop** | 低（拉低） | **高**（拉高） | 保持高 |

### 10.6 I2C_SendAck() — 主机发送应答信号

#### 代码实现

```c
void I2C_SendAck(void)
{
    /* ① 准备：SCL 低，SDA 高（释放） */
    SCL_LOW();
    SDA_HIGH();
    I2C_DELAY();

    /* ② SCL 低电平期间，SDA 拉低（设置 ACK 电平） */
    SDA_LOW();
    I2C_DELAY();

    /* ③ SCL 拉高，从设备采样 SDA（读到低电平 = ACK） */
    SCL_HIGH();
    I2C_DELAY();

    /* ④ SCL 拉低，采样结束 */
    SCL_LOW();
    I2C_DELAY();

    /* ⑤ 释放数据总线 */
    SDA_HIGH();
    I2C_DELAY();
}
```

### 10.7 I2C_SendNack() — 主机发送非应答信号

#### 与 ACK 的区别

NACK 时 SDA **保持高电平**（不拉低），其余步骤相同。

#### 代码实现

```c
void I2C_SendNack(void)
{
    /* ① 准备：SCL 低，SDA 高 */
    SCL_LOW();
    SDA_HIGH();
    I2C_DELAY();

    /* ② SDA 保持高（不做拉低操作 = NACK） */

    /* ③ SCL 拉高，从设备采样 SDA（读到高电平 = NACK） */
    SCL_HIGH();
    I2C_DELAY();

    /* ④ SCL 拉低，采样结束 */
    SCL_LOW();
    I2C_DELAY();

    /* ⑤ SDA 已经是高，无需额外操作（已释放） */
}
```

#### ACK 与 NACK 对比

| 操作 | 步骤② SDA | 采样结果 | 含义 |
|------|:---------:|:--------:|------|
| **SendAck** | **拉低** | 低电平 (0) | 应答：收到数据 |
| **SendNack** | **保持高** | 高电平 (1) | 非应答：不再接收 |

### 10.8 I2C_WaitAck() — 主机等待从设备应答

#### 与 SendAck/SendNack 的区别

| 对比项 | SendAck / SendNack | WaitAck |
|--------|-------------------|---------|
| 方向 | 主机**输出**到 SDA | 主机**读取** SDA |
| SDA 控制方 | **主机**控制 SDA | **从设备**控制 SDA |
| 操作前 | 主机拉低/保持高 SDA | 主机**释放** SDA（拉高） |
| 结果 | 无返回值 | **返回** ACK(0) 或 NACK(1) |

#### 代码实现

```c
uint8_t I2C_WaitAck(void)
{
    uint16_t ack;

    /* ① 释放数据总线（主机输出高阻态） */
    SCL_LOW();
    SDA_HIGH();
    I2C_DELAY();

    /* ② SCL 拉高，开始采样 */
    SCL_HIGH();
    I2C_DELAY();

    /* ③ 读取 SDA 电平 */
    ack = READ_SDA();    // 16 位值

    /* ④ SCL 拉低，采样结束 */
    SCL_LOW();
    I2C_DELAY();

    /* ⑤ 返回真值判断（非零=1=NACK，零=0=ACK） */
    return ack ? I2C_NACK : I2C_ACK;
}
```

### 10.9 I2C_SendByte() — 主机发送一个字节

#### 数据有效性时序回顾

```
SCL:  ─┐ ┌─┐ ┌─┐ ┌─┐ ┌─┐ ┌─┐ ┌─┐ ┌─┐ ┌─┐
    ───┘ └─┘ └─┘ └─┘ └─┘ └─┘ └─┘ └─┘ └─┘ └──
         D7  D6  D5  D4  D3  D2  D1  D0

SDA: ═══╤═══╤═══════╤═══╤═══════╤═══╤═════
        │ 1 │   0   │ 1 │   0   │ 1 │  0
        高位先行（MSB First）→ 逐位发送
```

#### 实现思路

| 要点 | 说明 |
|------|------|
| 循环次数 | **8 次**（一个字节 = 8 位） |
| 传输顺序 | **高位先行**（MSB First），先发 bit7，最后发 bit0 |
| 取最高位 | `byte & 0x80`（0x80 = 0b10000000） |
| 结果判断 | 非零 → 高电平(1)；零 → 低电平(0) |
| 移位操作 | 每次发完一位后 `byte <<= 1`（左移，保证下次仍取最高位） |

#### 代码实现

```c
void I2C_SendByte(uint8_t byte)
{
    uint8_t i;

    for (i = 0; i < 8; i++)
    {
        /* ① SCL 拉低，等待数据翻转 */
        SCL_LOW();
        SDA_LOW();      // 兼容起始信号后的初始状态
        I2C_DELAY();

        /* ② 取最高位，设置 SDA 电平 */
        if (byte & 0x80)
        {
            SDA_HIGH();  // 最高位为 1 → 拉高
        }
        else
        {
            SDA_LOW();   // 最高位为 0 → 拉低（可省略，①已拉低）
        }
        I2C_DELAY();

        /* ④ SCL 拉高，从设备采样 */
        SCL_HIGH();
        I2C_DELAY();

        /* ⑥ SCL 拉低，采样结束 */
        SCL_LOW();
        I2C_DELAY();

        /* ⑧ 左移一位，准备下一位（保证下次仍取最高位） */
        byte <<= 1;
    }
}
```

### 10.10 I2C_ReceiveByte() — 主机接收一个字节

#### 与 I2C_SendByte() 的核心区别

| 对比项 | SendByte | ReceiveByte |
|--------|----------|-------------|
| 数据方向 | 主机 → 从设备 | 从设备 → 主机 |
| SDA 控制 | **主机**控制 SDA | **从设备**控制 SDA |
| 主机操作 SDA | 根据数据拉高/拉低 | **不操作** SDA（释放） |
| 主机操作 SCL | 控制时钟翻转 | 控制时钟翻转 |
| 移位方向 | 左移取最高位发送 | 先左移，再存最低位 |
| 参数 | `uint8_t byte`（待发送） | 无参数 |
| 返回值 | 无 | `uint8_t data`（接收到的数据） |

#### 实现思路

```
初始化 data = 0
循环 8 次：
  ① SCL 拉低 → 等待从设备翻转 SDA
  ② SCL 拉高 → 开始采样
  ③ 读取 SDA 电平 → 存入 data 最低位
  ④ SCL 拉低 → 采样结束
返回 data
```

#### 关键点：先左移，再存入最低位

```
接收顺序：D7 → D6 → D5 → D4 → D3 → D2 → D1 → D0（高位先行）

第 1 次：先左移(data=00000000)，存 D7 到最低位 → 0000000[D7]
第 2 次：先左移              ，存 D6 到最低位 → 000000[D7][D6]
第 3 次：先左移              ，存 D5 到最低位 → 00000[D7][D6][D5]
...
第 8 次：先左移              ，存 D0 到最低位 → [D7][D6][D5][D4][D3][D2][D1][D0]
```

**为什么先左移再存入：**
```
❌ 错误方式：先存入再左移
  第 1 次：存 D7 → 0000000[D7] → 左移 → 000000[D7]0
  第 2 次：存 D6 → 000000[D6][D7]...混乱
  第 8 次：存 D0 后再左移 → D7 被移出字节，丢失！

✅ 正确方式：先左移再存入
  第 1 次：左移(00000000) → 存 D7 → 0000000[D7]
  第 2 次：左移(000000[D7]0) → 存 D6 → 000000[D7][D6]
  第 8 次：左移 → 存 D0 → [D7][D6]...[D1][D0] ✓
  （最后一次存入后不再移位，D0 在最低位，正确）
```

#### 代码实现

```c
uint8_t I2C_ReceiveByte(void)
{
    uint8_t i;
    uint8_t data = 0;    // 用于保存接收的数据，初始为 0

    for (i = 0; i < 8; i++)
    {
        /* 先左移，新数据存入最低位 */
        data <<= 1;

        /* ① SCL 拉低，等待从设备翻转 SDA */
        SCL_LOW();
        I2C_DELAY();

        /* ② SCL 拉高，开始采样 */
        SCL_HIGH();
        I2C_DELAY();

        /* ③ 读取 SDA 电平 */
        if (READ_SDA())     // 高电平 = 1
        {
            data |= 0x01;   // 最低位存 1
        }
        // 低电平 = 0：无需操作，左移后最低位已是 0

        /* ④ SCL 拉低，采样结束 */
        SCL_LOW();
        I2C_DELAY();
    }

    return data;
}
```

### 10.11 SendByte 与 ReceiveByte 的移位对比

| 对比项 | SendByte | ReceiveByte |
|--------|----------|-------------|
| 移位时机 | **发完后**左移 | **读取前**左移 |
| 取位位置 | 每次取**最高位** (`byte & 0x80`) | 每次存入**最低位** (`data \|= 0x01`) |
| 移位对象 | 参数 byte（值传递，可直接改） | 局部变量 data |
| 移位方向 | 左移 | 左移 |
| 移位目的 | 原最高位发出后，下一位成为新最高位 | 已存数据左移，为新位腾出最低位 |

```
SendByte:    [D7 D6 D5 D4 D3 D2 D1 D0] → 取 D7 → 左移 → [D6 D5 D4 D3 D2 D1 D0 0]
                                                  ↑
                                          每次取最高位发送

ReceiveByte: [0  0  0  0  0  0  0  0 ] → 左移 → [0 0 0 0 0 0 0 0] → 存 D7 → [0 0 0 0 0 0 0 D7]
                                                  ↑
                                          先左移，再存最低位
```

### 10.12 所有协议函数的共同模式

每个 I²C 协议函数都遵循相同的时钟控制模式：

```
通用模式：
  ① SCL 低电平期间 → SDA 设置目标电平（或释放）
  ② I2C_DELAY()     → 保持一段时间
  ③ SCL 拉高        → 开始采样
  ④ I2C_DELAY()     → 采样保持
  ⑤ SCL 拉低        → 采样结束
  ⑥ I2C_DELAY()     → 准备下一位
```

### 11.3 EEPROM_WriteByte() — 写入一个字节

### 11.4 EEPROM_ReadByte() — 读取一个字节（假写真读）

#### 时序图

```
假写阶段：
[S] → [0xA0(W)] → ACK → [内部地址] → ACK

真读阶段：
[Sr] → [0xA1(R)] → ACK → [从设备发送数据] → 主机 NACK → [P]
```

#### 步骤对照

| 步骤 | 操作 | 代码 | 说明 |
|:----:|------|------|------|
| ① | 起始信号 | `I2C_Start();` | 开始假写阶段 |
| ② | 发送写地址 0xA0 | `I2C_SendByte(W_ADDR);` | 假写：用写地址保持总线控制权 |
| ③ | 等待应答 | `I2C_WaitAck();` | |
| ④ | 发送内部地址 | `I2C_SendByte(inner_addr);` | 写入地址计数器 |
| ⑤ | 等待应答 | `I2C_WaitAck();` | 假写阶段结束 |
| ⑥ | **重复起始** | `I2C_Start();` | Sr：不发 P，保持总线占用 |
| ⑦ | 发送读地址 0xA1 | `I2C_SendByte(R_ADDR);` | 真读：切换为读方向 |
| ⑧ | 等待应答 | `I2C_WaitAck();` | 总线控制权交给从设备 |
| ⑨ | 读取数据 | `byte = I2C_ReceiveByte();` | 从设备发送数据 |
| ⑩ | 发送非应答 | `I2C_SendNack();` | 只读1字节，告诉从设备"够了" |
| ⑪ | 停止信号 | `I2C_Stop();` | 结束通讯 |
| — | 返回数据 | `return byte;` | |

#### 代码实现

```c
uint8_t EEPROM_ReadByte(uint8_t inner_addr)
{
    uint8_t byte;

    // ===== 假写阶段：设置地址计数器 =====

    // ① 起始信号
    I2C_Start();

    // ② 发送写地址（假写，保持主设备总线控制权）
    I2C_SendByte(W_ADDR);

    // ③ 等待应答
    I2C_WaitAck();

    // ④ 发送内部存储地址（写入地址计数器）
    I2C_SendByte(inner_addr);

    // ⑤ 等待应答
    I2C_WaitAck();

    // ===== 真读阶段：从指定地址读取数据 =====

    // ⑥ 重复起始信号（Sr，不发 P）
    I2C_Start();

    // ⑦ 发送读地址（切换为读方向）
    I2C_SendByte(R_ADDR);

    // ⑧ 等待应答（总线控制权交给从设备）
    I2C_WaitAck();

    // ⑨ 读取一个字节数据
    byte = I2C_ReceiveByte();

    // ⑩ 主机发送非应答（只读1字节，不需要更多数据）
    I2C_SendNack();

    // ⑪ 停止信号
    I2C_Stop();

    return byte;
}
```

## 十二、连续读写多个字节

### 12.1 EEPROM_WriteBytes() — 连续写入多个字节（页写）

#### 时序图

```
[S] → [0xA0(W)] → ACK → [内部地址] → ACK
  → [数据1] → ACK → [数据2] → ACK → ... → [数据N] → ACK
  → [P]
                                              等待 5ms
```

#### 与单字节写入的区别

| 对比项 | WriteByte | WriteBytes |
|--------|-----------|------------|
| 数据数量 | 1 个字节 | size 个字节 |
| 数据发送 | 直接发送 1 次 | **for 循环** size 次 |
| ACK 次数 | 3 次 | 2 + size 次 |
| 结束方式 | 数据 ACK 后 P | 循环结束后 P |
| 写入周期 | 5ms | 5ms |

#### 代码实现

```c
void EEPROM_WriteBytes(uint8_t inner_addr, uint8_t *data, uint8_t size)
{
    uint8_t i;

    // ① 起始信号
    I2C_Start();

    // ② 发送写地址
    I2C_SendByte(W_ADDR);

    // ③ 等待应答
    I2C_WaitAck();

    // ④ 发送内部存储地址（写入起始位置）
    I2C_SendByte(inner_addr);

    // ⑤ 等待应答
    I2C_WaitAck();

    // ⑥~⑦ 循环发送数据
    for (i = 0; i < size; i++)
    {
        I2C_SendByte(data[i]);    // 发送第 i 个字节
        I2C_WaitAck();            // 等待应答
    }

    // ⑧ 停止信号
    I2C_Stop();

    // ⑨ 等待写入周期（5ms）
    Delay_ms(5);
}
```

### 12.2 EEPROM_ReadBytes() — 连续读取多个字节

#### 时序图

```
假写：
[S] → [0xA0(W)] → ACK → [起始内部地址] → ACK

真读：
[Sr] → [0xA1(R)] → ACK
  → [数据1] → 主机 ACK
  → [数据2] → 主机 ACK
  → ...
  → [数据N] → 主机 NACK
  → [P]
```

#### 与单字节读取的区别

| 对比项 | ReadByte | ReadBytes |
|--------|----------|-----------|
| 数据数量 | 1 个字节 | size 个字节 |
| 数据接收 | 接收 1 次 | **for 循环** size 次 |
| 存储方式 | `return byte` | 存入 `buffer[i]` |
| 应答方式 | 直接 NACK | 前 N-1 次 **ACK**，最后 1 次 **NACK** |
| 返回值 | uint8_t | void（数据存在 buffer 中） |

#### 代码实现

```c
void EEPROM_ReadBytes(uint8_t inner_addr, uint8_t *buffer, uint8_t size)
{
    uint8_t i;

    // ===== 假写阶段 =====

    // ① 起始信号
    I2C_Start();

    // ② 发送写地址（假写）
    I2C_SendByte(W_ADDR);

    // ③ 等待应答
    I2C_WaitAck();

    // ④ 发送内部存储地址（写入地址计数器）
    I2C_SendByte(inner_addr);

    // ⑤ 等待应答
    I2C_WaitAck();

    // ===== 真读阶段 =====

    // ⑥ 重复起始信号
    I2C_Start();

    // ⑦ 发送读地址（真读）
    I2C_SendByte(R_ADDR);

    // ⑧ 等待应答
    I2C_WaitAck();

    // ⑨ 循环读取数据
    for (i = 0; i < size; i++)
    {
        buffer[i] = I2C_ReceiveByte();

        if (i < size - 1)
        {
            I2C_SendAck();      // 非最后一字节：ACK，告诉从设备继续发
        }
        else
        {
            I2C_SendNack();     // 最后一字节：NACK，告诉从设备停止发
        }
    }

    // ⑩ 停止信号
    I2C_Stop();
}
```

#### ACK / NACK 判断逻辑

```
i = 0: buffer[0] = 读取 → ACK  （还有更多数据）
i = 1: buffer[1] = 读取 → ACK  （还有更多数据）
...
i = size-2: buffer[size-2] = 读取 → ACK  （还有1个）
i = size-1: buffer[size-1] = 读取 → NACK （最后一个，停止）

判断条件：i < size - 1 → ACK；i == size - 1 → NACK
```

### 12.3 连续写入注意事项

| 注意点 | 说明 |
|--------|------|
| 页大小限制 | 单次最多写入 **16 字节**（一页） |
| 不可跨页 | 地址到页末后自动绕回页首 |
| 跨页解决方案 | 分多次调用 `WriteBytes`，每次指定新的 `inner_addr` |
| 写入周期 | 每次写入后等待 **5ms** |
| 内部地址自动递增 | AT24C02 每写入一个字节后地址计数器 +1 |

#### 跨页写入示例

```c
// 向地址 0x0E 开始写入 6 字节（跨越页边界）

// 第一次：写前 2 字节到第 0 页末尾（0x0E, 0x0F）
EEPROM_WriteBytes(0x0E, data, 2);

// 第二次：写后 4 字节到第 1 页开头（0x10, 0x11, 0x12, 0x13）
EEPROM_WriteBytes(0x10, data + 2, 4);
```

### 12.4 连续读取注意事项

| 注意点 | 说明 |
|--------|------|
| 无页大小限制 | 读取可连续，地址计数器自动递增，不绕回 |
| 必须发 NACK | 最后一个字节读完后发 NACK，否则从设备继续发送 |
| 可跨页读取 | 地址自动递增可跨越页边界 |

### 14.4 页写绕回现象分析

#### 写入 21 字节的实际行为

```
AT24C02 第 0 页（地址 0x00 ~ 0x0F，共 16 字节）：

写入顺序：
  第 1 字节：'1'  → 地址 0x00
  第 2 字节：'2'  → 地址 0x01
  ...
  第16 字节：'f'  → 地址 0x0F
  ─────────── 页写满，地址绕回 ───────────
  第17 字节：'j'  → 地址 0x00（覆盖 '1'）
  第18 字节：'h'  → 地址 0x01（覆盖 '2'）
  第19 字节：'i'  → 地址 0x02（覆盖 '3'）
  第20 字节：'j'  → 地址 0x03（覆盖 '4'）
  第21 字节：'k'  → 地址 0x04（覆盖 '5'）

最终内存状态：
  地址 0x00~0x04：j h i j k（被覆盖）
  地址 0x05~0x0F：6 7 8 9 0 a b c d e f（原数据保留）
```

## 十七、关键记忆口诀

| 口诀 | 含义 |
|------|------|
| **SCL 高时看 SDA** | 高电平采样数据或检测起始/停止跳变 |
| **SCL 低时改 SDA** | 低电平时才能翻转数据线电平 |
| **起始：高拉低** | SCL 高时 SDA 下降沿 |
| **停止：低拉高** | SCL 高时 SDA 上升沿 |
| **ACK 是 0，NACK 是 1** | 应答拉低，非应答保持高 |
| **高位先行 MSB** | I²C 先发最高位，与串口相反 |
| **假写真读** | 用写地址设置计数器，再用读地址读数据 |
| **ACK 继续，NACK 停止** | 连续读取时主机控制从设备是否继续发送 |

## 十九、STM32 硬件 I²C 外设

### 19.1 软件模拟 I²C 的问题

| 问题 | 说明 |
|------|------|
| 代码量大 | 每个协议步骤都需要手动实现（拉高拉低 + 延迟） |
| 容易出错 | 一个时序细节写错，整个通讯失败 |
| 时序不精确 | Delay_us(10) 是估算值，未考虑上升/下降沿时间 |
| CPU 资源浪费 | 每个电平翻转都由 CPU 执行指令完成 |
| 效率低 | 所有协议细节由软件控制，CPU 被大量占用 |

> **解决方案**：使用 STM32 内置的 **I²C 硬件外设**，由硬件电路自动处理协议时序。

📄 [原文07: STM32硬件I2C外设介绍](../raw/嵌入式开发/I2C通信协议/07_STM32硬件I2C外设介绍.md)

### 19.2 硬件 I²C vs 软件模拟 I²C

| 对比项 | 软件模拟 | 硬件 I²C |
|--------|----------|----------|
| 时序控制 | 软件 Delay 粗略控制 | **硬件精确控制**（时钟电路自动生成） |
| CPU 占用 | 高（每步都需 CPU 执行） | **低**（硬件自动处理协议） |
| 代码量 | 大（手动实现每个步骤） | **小**（配置寄存器后自动运行） |
| 可靠性 | 依赖软件逻辑，易出错 | **硬件保证时序正确** |
| 灵活性 | 可用任意 GPIO 引脚 | 受限于固定复用引脚 |
| 速度控制 | 不精确 | 精确（CCR 寄存器分频） |
| 应答处理 | 软件读取 SDA 电平判断 | **硬件自动捕获应答信号** |
| 适用场景 | 学习理解协议 / 引脚受限 | **正式开发推荐** |

### 19.4 I²C 硬件外设功能框图

#### 核心模块说明

| 模块 | 功能 | 对应寄存器 |
|------|------|-----------|
| **数据移位寄存器** | 串并转换（发送：并→串；接收：串→并） | — |

### 19.5 数据收发原理

#### 发送过程（主设备发送）

```
CPU 写入 DR（并行数据）
    ↓
数据移位寄存器加载 DR 数据
    ↓
最高位(MSB)先发 → 依次左移 → 最低位(LSB)最后发
    ↓
每一位通过 SDA 线发出（SCL 时钟同步）
    ↓
发完 8 位 → 等待 ACK（硬件自动检测）
    ↓
状态寄存器更新标志位
```

#### 接收过程（主设备接收）

```
SDA 线上的串行数据逐位进入移位寄存器
    ↓
从最低位开始接收，每次左移一位
（先收到的位逐渐移到最高位位置）
    ↓
收完 8 位 → 整体传送到 DR
    ↓
CPU 从 DR 读取数据（并行）
    ↓
主机发送 ACK/NACK（硬件控制）
```

#### 与串口的对比

| 对比项 | 串口 (USART) | I²C 硬件外设 |
|--------|-------------|-------------|
| 数据寄存器 | TDR（发）+ RDR（收）分开 | **DR 合一**（半双工，分时复用） |
| 移位寄存器 | TX 移位 + RX 移位分开 | **一个移位寄存器**（分时复用） |
| 双工 | 全双工（独立收发通路） | 半双工（共用一条数据通路） |
| 波特率控制 | BRR 寄存器 | **CCR 寄存器**（时钟分频） |
| 状态标志 | TXE / RXNE | **TXE / RXNE**（类似） |
| 应答机制 | 无 | **有**（硬件自动检测 ACK/NACK） |

## 二十、硬件 I²C 寄存器实现

### 20.1 寄存器配置基础

需要配置的 4 个核心寄存器及其配置时机：

| 寄存器 | 名称 | 配置内容 | 配置时机 |
|--------|------|----------|----------|
| **CR1** | 控制寄存器 1 | 模式选择(SMBus/I²C)、ACK、START、STOP、PE 使能 | PE 位**最后**使能 |
| **CR2** | 控制寄存器 2 | 输入时钟频率 FREQ[5:0] | PE 使能前 |
| **CCR** | 时钟控制寄存器 | SCL 频率分频系数、标准/快速模式选择 | PE 使能前 |
| **TRISE** | 上升时间寄存器 | SCL 最大上升沿时间 | PE 使能前 |

CCR 计算公式（标准模式）：

- `CCR = thigh × f_PCLK1`，其中 thigh = 5μs（100kHz 下高低电平各半）
- 36MHz 输入时钟：CCR = 5μs × 36MHz = **180**

TRISE 计算公式：

- `TRISE = (tr / T_PCLK1) + 1`，其中 tr = 1μs（标准模式最大上升时间）
- 36MHz 输入时钟：TRISE = 36 + 1 = **37**

> **必须先配好 CR2、CCR、TRISE 再开启 PE**，PE 使能后这三个寄存器不可修改。

📄 [原文08: STM32硬件I2C寄存器配置详解](../raw/嵌入式开发/I2C通信协议/08_STM32硬件I2C寄存器配置详解.md)

### 20.2 关键寄存器详解

#### CR1 控制位

| 位 | 名称 | 功能 | 特性 |
|:--:|------|------|------|
| 0 | **PE** | 外设使能 | 最后置 1 |
| 8 | **START** | 产生起始条件 | 写 1 产生，硬件自动清除 |
| 9 | **STOP** | 产生停止条件 | **配置项**，字节传输结束后生效 |
| 10 | **ACK** | 应答使能 | **配置项**：1=收到字节后发 ACK，0=NACK |

控制位特性对比：

| 控制位 | 类型 | 生效时机 | 是否需要等待 |
|--------|------|----------|:------------:|
| **START** | 即时触发 | 重复尝试直到总线空闲 | 需等待 SB |
| **STOP** | 配置项 | 当前字节传输结束后 | 无需等待 |
| **ACK** | 配置项 | 收到字节后自动执行 | 无需等待 |

#### SR1 状态标志位

| 位 | 名称 | 含义 | 使用场景 | 清除方式 |
|:--:|------|------|----------|----------|
| 0 | **SB** | 起始位已发出 | Start 后等待 | 读 SR1 + 写 DR |
| 1 | **ADDR** | 地址已发送/匹配 | SendAddr 后等待 | 读 SR1 + 读 SR2 |
| 2 | **BTF** | 字节传输完成（含 ACK） | SendByte 后等待 | 读/写 DR 或 START/STOP |
| 6 | **RXNE** | 接收数据寄存器非空 | ReceiveByte 前等待 | 读 DR |
| 7 | **TXE** | 发送数据寄存器为空 | SendByte 前等待 | 写 DR |

#### 主模式发送/接收流程

```
发送：START → 等SB → 发地址 → 等ADDR → 发数据 → 等TXE → 写DR → 等BTF
接收：START → 等SB → 发地址 → 等ADDR → 等RXNE → 读DR
```

#### 主模式 vs 从模式

| 对比项 | 主模式 | 从模式 |
|--------|--------|--------|
| 时钟控制 | **主设备控制 SCL** | 被动接收 SCL |
| 起始/停止信号 | **主设备发出** S 和 P | 检测 S 和 P |
| 检测标志 | **SB**（起始已发） | **STOPF**（收到停止） |
| 自身地址 | **不需要**配 | 需要配 OAR1/OAR2 |

> 每个等待都应加入超时机制（如 `timeout = 0xFFFF` 递减），防止 while 循环卡死。

### 20.3 初始化配置

#### 配置流程

```
① 开启时钟（GPIOB + I²C2）
② 配置 GPIO（PB10/PB11 复用开漏输出 50MHz）
③ CR2 = 36（输入时钟频率 36MHz）
④ CCR = 180（标准模式，分频系数）
⑤ TRISE = 37（最大上升时间）
⑥ CR1：I²C 模式 + ACK 使能
⑦ CR1：PE 外设使能（最后开启）
```

| 项目 | 说明 |

| 模式 | **复用开漏输出** 50MHz（CNF=11, MODE=11） |

| 参数 | 数值 | 来源 |
|------|------|------|
| APB1 时钟 | 36 MHz | 时钟树配置 |
| CR2 FREQ | **36** | APB1 = 36MHz |
| CCR | **180** | 5μs × 36MHz |
| TRISE | **37** | 36 + 1 |

### 20.4 起始/停止/应答函数

#### 函数语义变化

| 软件模拟 | 硬件 I²C | 本质变化 |
|----------|----------|----------|
| `I2C_Start()` → 手动拉高拉低 SDA | `I2C_Start()` → CR1.START=1，等 SB=1 | 从手动操作变为硬件自动 |
| `I2C_Stop()` → 立即执行 | `I2C_Stop()` → CR1.STOP=1，延迟生效 | 从即时变为**配置项** |
| `I2C_SendAck()` → 立即拉低 SDA | `I2C_EnableAck()` → CR1.ACK=1 | 从输出操作变为**配置项** |
| `I2C_SendNack()` → 保持 SDA 高 | `I2C_DisableAck()` → CR1.ACK=0 | 从输出操作变为**配置项** |
| `I2C_WaitAck()` → 读 SDA 电平 | **删除**（融入 SendByte/SendAddr） | 等待 BTF/ADDR 即含应答检测 |

#### WaitAck 被删除的原因

硬件 I²C 中，应答检测已融入数据收发函数：

- 发送字节后等待 **BTF** = 1 → BTF 置位说明字节传输完成（已收到 ACK）
- 发送地址后等待 **ADDR** = 1 → ADDR 置位说明地址发送成功（已收到 ACK）
- BTF/ADDR 标志位本身就包含了"收到应答"的含义

### 20.5 数据收发实现

#### 发送一个字节（I2C_SendByte）

```
① 等待 TXE = 1（发送缓冲区空，可写入新数据）
② 写入数据到 DR
③ 等待 BTF = 1（字节传输完成，已收到 ACK）
④ 返回 OK 或 TIMEOUT
```

| 等待标志 | 含义 | 清除方式 |
|----------|------|----------|
| TXE | DR 数据已移入移位寄存器 | 写 DR |
| BTF | 字节传输完成+收到 ACK | 读/写 DR 或 START/STOP |

#### 接收一个字节（I2C_ReceiveByte）

```
① 等待 RXNE = 1（接收缓冲区非空，可读取数据）
② 读取 DR，返回数据
```

| 等待标志 | 含义 | 清除方式 |
|----------|------|----------|
| RXNE | 移位寄存器数据已传送到 DR | 读 DR |

#### 发送设备地址（I2C_SendAddr）

```
① 直接写入 DR（起始后 DR 为空，无需等 TXE）
② 等待 ADDR = 1（地址已发送，收到 ACK）
③ 读 SR2 清除 ADDR 标志
④ 返回 OK 或 TIMEOUT
```

> SendAddr 与 SendByte 的关键区别：SendAddr 等待 **ADDR** 标志并需读 **SR2** 清除；SendByte 等待 **BTF** 标志。

#### 所有函数操作流程

```
I2C_Start():    CR1.START=1 → [等 SB=1] → 返回 OK/TIMEOUT
I2C_Stop():     CR1.STOP=1（延迟生效）
I2C_EnableAck():  CR1.ACK=1
I2C_DisableAck(): CR1.ACK=0
I2C_SendAddr(): DR=addr → [等 ADDR=1] → 读SR2清除 → 返回 OK/TIMEOUT
I2C_SendByte(): [等 TXE=1] → DR=byte → [等 BTF=1] → 返回 OK/TIMEOUT
I2C_ReceiveByte(): [等 RXNE=1] → 返回 DR/TIMEOUT
```

### 20.6 AT24C02 接口层适配

#### 头文件（m24c02.h）— 无需修改

接口层函数声明与软件模拟版本完全一致，主函数调用方式也不变。

#### 核心改动：调用方式变化

| 软件模拟调用 | 硬件 I²C 调用 | 变化 |
|-------------|--------------|------|
| `I2C_SendByte(0xA0)` + `I2C_WaitAck()` | `I2C_SendAddr(0xA0)` | **合并**为专用函数 |
| `I2C_SendByte(data)` + `I2C_WaitAck()` | `I2C_SendByte(data)` | 内置 BTF 等待 |
| `I2C_SendAck()` | `I2C_EnableAck()` | 改名（配置项） |
| `I2C_SendNack()` | `I2C_DisableAck()` | 改名（配置项） |

#### 读取最后一个字节的特殊处理

硬件 I²C 中 ACK、NACK、STOP 都是**配置项**，设置后在**下一个字节传输完成后**才生效，因此必须**先设置再读取**：

```
✅ 正确顺序（硬件 I²C 思路）：
  DisableAck()     ← 设置：下一个字节收到后不返回 ACK
  I2C_Stop()       ← 设置：下一个字节传输结束后发出停止
  byte = ReceiveByte()  ← 读取字节 → 硬件自动执行 NACK + STOP
```

#### 连续读取中的 ACK/NACK/STOP 设置

| 情况 | 设置行为 |
|------|----------|
| 非最后一字节（`i < size-1`） | `EnableAck()` → 收到字节后返回 ACK |
| 最后一字节（`i == size-1`） | `DisableAck()` + `Stop()` → 收到字节后发 NACK + STOP |

#### 接口层函数步骤数对比

| 函数 | 软件模拟版本 | 硬件 I²C 版本 |
|------|-------------|--------------|
| **WriteByte** | 8 步（含 3 次 WaitAck） | **5 步** |
| **ReadByte** | 11 步（含 3 次 WaitAck） | **8 步** |
| **WriteBytes** | 8 步（含 N+2 次 WaitAck） | **5 步** |
| **ReadBytes** | 11 步（含 N+2 次 WaitAck） | **8 步** |

#### 三种 I²C 实现方式对比

| 对比项 | 软件模拟 I²C | 硬件 I²C（寄存器） | 硬件 I²C（HAL 库） |
|--------|-------------|-------------------|-------------------|
| GPIO 模式 | 通用开漏输出 | 复用开漏输出 | CubeMX 配置 |
| 时序控制 | 手动 Delay | 硬件自动（CCR） | 硬件自动 |
| CPU 占用 | 高 | 低 | 低 |
| 代码量 | 大 | 中 | 小 |
| 引脚灵活性 | 任意 GPIO | 固定复用引脚 | 固定复用引脚 |
| 可靠性 | 依赖软件 | 硬件保证 | 硬件保证 |
