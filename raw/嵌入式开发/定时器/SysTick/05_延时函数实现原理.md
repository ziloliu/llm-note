# 延时函数（Delay）实现原理详解 笔记

---

## 一、延时函数层次结构

```
Delays(sec)    ← 秒级延时
    └── 调用 Delay_ms(1000) × sec 次

Delay_ms(ms)   ← 毫秒级延时
    └── 调用 Delay_us(1000) × ms 次

Delay_us(us)   ← 微秒级延时（基础实现）
    └── 直接操作 SysTick 定时器
```

---

## 二、Delay_us() — 微秒级延时（轮询方式）

### 2.1 核心原理

```
SysTick 时钟频率 = 72 MHz
每次滴答 = 1/72 μs
72 次滴答 = 1 μs

Delay_us(us) 所需滴答次数 = us × 72
```

### 2.2 代码实现

```c
void Delay_us(uint16_t us)
{
    SysTick->LOAD = 72 * us;           // 重装载值
    SysTick->VAL  = 0;                  // 清零当前值
    SysTick->CTRL = 0x05;               // 101: 72MHz + 无中断 + 使能

    while (!(SysTick->CTRL & (1 << 16)));  // 等待 COUNTFLAG = 1

    SysTick->CTRL = 0x00;               // 关闭定时器
}
```

### 2.3 CTRL = 0x05 详解

| 位 | 值 | 含义 |
|:--:|:--:|------|
| bit2 (CLKSOURCE) | **1** | AHB 时钟 72MHz（不分频） |
| bit1 (TICKINT) | **0** | 不产生中断 |
| bit0 (ENABLE) | **1** | 使能定时器 |

```
0x05 = 0b101 = ENABLE(1) + TICKINT(0) + CLKSOURCE(1)
```

## 三、LOAD 值分析

### 3.1 数据类型与范围约束

```
参数类型：uint16_t us
  → us 最大值 = 65,535

LOAD 最大值 = 65,535 × 72 = 4,718,520

24 位最大值 = 2^24 - 1 = 16,777,215

4,718,520 < 16,777,215  →  ✅ 不会溢出
```

### 3.2 数据类型选择原因

```
为什么用 uint16_t 而不是 uint32_t？

  uint16_t 最大 65,535 → 65,535 × 72 = 4.7M → 不超 24 位限制 ✅
  uint32_t 最大 4,294,967,295 → 会远超 24 位限制 ❌

用 uint16_t 自然保证了不会超出 SysTick 的 24 位范围
```

## 四、轮询方式 vs 中断方式的 LOAD 值差异

### 4.1 轮询方式（Delay_us）

```c
// 目标：延迟 1ms = 72000 次滴答
SysTick->LOAD = 72000;      // 不需要减 1 ✅
```

### 4.2 中断方式（SysTick 中断）

```c
// 目标：每 1ms 产生一次中断
SysTick->LOAD = 71999;      // 需要减 1 ✅
```

### 4.3 为什么有差异

```
轮询方式：
  LOAD = 72000
  72000 → 71999 → ... → 1 → 0
                            ↓
                     检测到 COUNTFLAG = 1，立即退出
  实际经过 72000 次滴答 ✅
  只要 COUNTFLAG 为 1 就退出，时机精确

中断方式：
  LOAD = 72000（不减 1 会怎样？）
  72000 → 71999 → ... → 1 → 0 → ？？→ 下一个时钟周期触发中断
  实际经过 72001 次滴答 ❌ 多了 1 次

  LOAD = 71999（减 1）
  71999 → 71998 → ... → 1 → 0 → ？？→ 下一个时钟周期触发中断
  实际经过 72000 次滴答 ✅ 正确
```

### 4.4 差异原因

| 方式 | 检测/触发时机 | 检测点 |
|------|-------------|--------|
| **轮询** | 读到 COUNTFLAG=1 时**立即**退出 | 计数器到达 0 的**同一**时钟周期 |
| **中断** | 下一个时钟周期才进入中断 | 计数器到达 0 的**下一**时钟周期 |

> 中断方式比轮询方式多经过 1 个时钟周期，因此 LOAD 需要减 1 补偿。

## 五、Delay_ms() — 毫秒级延时

### 5.1 为什么用循环而不是乘法

```c
// ❌ 直接乘法可能溢出
void Delay_ms(uint16_t ms)
{
    Delay_us(1000 * ms);    // 1000 × ms 可能超出 uint16_t 范围
}
// uint16_t 最大 65535
// 1000 × 65 = 65000 → ✅ 安全
// 1000 × 66 = 66000 → ❌ 溢出！
```

```c
// ✅ 循环方式避免溢出
void Delay_ms(uint16_t ms)
{
    while (ms--)
    {
        Delay_us(1000);     // 每次只延迟 1ms，参数不会溢出
    }
}
```

### 5.2 代码实现

```c
void Delay_ms(uint16_t ms)
{
    while (ms--)
    {
        Delay_us(1000);     // 每次延迟 1000μs = 1ms
    }
}
```

## 七、轮询延时的问题

### 7.1 轮询延时的缺点

| 问题 | 说明 |
|------|------|
| **阻塞 CPU** | while 循环期间 CPU 无法执行其他任务 |
| **中断冲突** | 如果中断中也调用 Delay，可能卡死 |
| **实时性差** | 其他任务必须等延时完成才能执行 |
| **效率低** | CPU 在空循环中浪费时间 |

### 7.2 中断嵌套卡死示例

```
主函数：
  Delay_ms(1000);       ← 正在轮询等待 COUNTFLAG
      ↓
  某个中断触发
      ↓
  中断服务函数中：
      Delay_ms(100);    ← 又要操作 SysTick，可能冲突！
```

## 八、HAL 库延时函数（中断方式）

### 8.1 HAL_Delay() 实现原理

```c
// HAL 库的延时函数（__weak 可重写）
__weak void HAL_Delay(uint32_t Delay)
{
    uint32_t tickstart = HAL_GetTick();   // 获取当前滴答值
    uint32_t wait = Delay;

    // 防止最小延迟为 1（确保至少等待一个滴答）
    if (wait < HAL_MAX_DELAY)
    {
        wait += uwTickFreq;               // wait += 1
    }

    // 等待直到经过了足够的滴答数
    while ((HAL_GetTick() - tickstart) < wait)
    {
        // 空循环，但 SysTick 中断正常工作
        // uwTick 在中断中持续递增
    }
}
```

### 8.2 工作原理图

```
uwTick 全局变量：持续递增（每 1ms +1）
     │
     ├── 调用 HAL_Delay(500) 时：
     │     tickstart = uwTick（假设为 100）
     │     wait = 500
     │     while ((uwTick - 100) < 500)
     │         → 等待 uwTick 达到 600
     │         → uwTick 由 SysTick 中断驱动，不停增长
     │         → 退出条件：uwTick = 600
     │
     └── 期间 SysTick 中断正常工作：
           uwTick: 100 → 101 → 102 → ... → 600
           每 1ms 中断一次，每次 uwTick += 1
```

### 8.3 与轮询方式的关键区别

| 对比项 | 轮询方式 | HAL_Delay（中断方式） |
|--------|----------|----------------------|
| 计时方式 | 操作 SysTick 硬件 | 读取全局变量 uwTick |
| SysTick 占用 | 函数内独占 | 与系统共享 |
| 是否阻塞 | 完全阻塞 | 阻塞但中断正常工作 |
| 能否嵌套 | 容易卡死 | 相对安全 |
| 精确度 | 硬件计时 | 软件计时（受中断延迟影响） |

## 九、用中断方式重写自定义延时函数

### 9.1 设计思路

```c
// 全局变量（类似 HAL 库的 uwTick）
volatile uint32_t sys_tick = 0;

// SysTick 初始化：每 1ms 中断
void SysTick_Init(void)
{
    SysTick->LOAD = 71999;
    SysTick->CTRL = 0x07;               // 111: 使能 + 中断 + 72MHz
}

// 中断服务函数
void SysTick_Handler(void)
{
    sys_tick++;                         // 每 1ms 加 1
}

// 获取当前滴答值
uint32_t GetTick(void)
{
    return sys_tick;
}

// 毫秒延时（中断方式）
void Delay_ms(uint32_t ms)
{
    uint32_t start = GetTick();
    while ((GetTick() - start) < ms)
    {
        // 等待，SysTick 中断驱动 sys_tick 递增
    }
}

// 微秒延时（可选：混合方式）
void Delay_us(uint16_t us)
{
    SysTick->LOAD = 72 * us;
    SysTick->VAL  = 0;
    SysTick->CTRL = 0x05;               // 轮询方式
    while (!(SysTick->CTRL & (1 << 16)));
    SysTick->CTRL = 0x00;
}
```

## 十、三种延时实现方式对比

| 对比项 | 轮询方式 | 中断方式（自定义） | HAL_Delay |
|--------|----------|-------------------|-----------|
| 计时源 | SysTick 硬件 COUNTFLAG | 全局变量 sys_tick | 全局变量 uwTick |
| SysTick 配置 | 无中断（101） | 有中断（111） | 有中断（111） |
| 是否阻塞 | 完全阻塞 | 阻塞但中断工作 | 阻塞但中断工作 |
| 能否嵌套 | 容易卡死 | 较安全 | 较安全 |
| 精确度 | 最高（硬件） | 较高 | 较高（受中断延迟影响） |
| 实现复杂度 | 最简单 | 中等 | HAL 自动 |

## 十一、完整 Delay 代码回顾

### 11.1 delay.h

```c
#ifndef __DELAY_H
#define __DELAY_H

#include "stm32f10x.h"

void Delay_us(uint16_t us);
void Delay_ms(uint16_t ms);
void Delays(uint16_t s);

#endif
```

### 11.2 delay.c（轮询方式）

```c
#include "delay.h"

/**
 * @brief  微秒级延时（轮询方式）
 * @param  us: 延时微秒数（16位，最大65535μs ≈ 65ms）
 */
void Delay_us(uint16_t us)
{
    SysTick->LOAD = 72 * us;                        // 装载值
    SysTick->VAL  = 0;                               // 清零
    SysTick->CTRL = 0x05;                            // 101: 72MHz + 无中断 + 使能

    while (!(SysTick->CTRL & (1 << 16)));            // 等待 COUNTFLAG

    SysTick->CTRL = 0x00;                            // 关闭
}

/**
 * @brief  毫秒级延时
 * @param  ms: 延时毫秒数
 */
void Delay_ms(uint16_t ms)
{
    while (ms--)
    {
        Delay_us(1000);
    }
}

/**
 * @brief  秒级延时
 * @param  s: 延时秒数
 */
void Delays(uint16_t s)
{
    while (s--)
    {
        Delay_ms(1000);
    }
}
```
