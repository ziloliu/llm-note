# 输入捕获代码优化 — 事件先于中断的核心原理 笔记

---

## 一、初始版本（带全局变量）

### 1.1 实现思路

```c
volatile uint16_t cycle = 0;    // 保存周期
volatile uint8_t  count = 0;    // 上升沿计数

void TIM4_IRQHandler(void)
{
    if (TIM4->SR & TIM_SR_CC1IF)
    {
        TIM4->SR &= ~TIM_SR_CC1IF;
        count++;

        if (count == 1)          // 第 1 次上升沿
            TIM4->CNT = 0;       // 清零计数器
        else if (count == 2)     // 第 2 次上升沿
        {
            cycle = TIM4->CCR1;  // 读取周期
            count = 0;           // 重置
        }
    }
}
```

### 1.2 工作流程

```
第 1 上升沿 → count=1 → CNT=0
第 2 上升沿 → count=2 → cycle=CCR1=10000 → count=0
第 3 上升沿 → count=1 → CNT=0
第 4 上升沿 → count=2 → cycle=CCR1=10000 → count=0
...
```

## 二、直接用 CCR1 出错的原因

### 2.1 问题现象

```c
// 直接返回 CCR1，去掉 cycle 全局变量
double TIM4_GetPWMPeriod(void)
{
    return (double)TIM4->CCR1 / 1000.0;
}

// 结果：周期在 10ms 和 20ms 之间交替！
```

### 2.2 根本原因

```
时间线：

① 第 1 上升沿：count=1 → CNT=0
   CNT 开始计数：0,1,2,...,9999

② 第 2 上升沿：
   硬件：CCR1 = CNT = 10000 ✅（正确）
   中断：cycle = CCR1 → count=0

③ 第 3 上升沿：
   关键问题：CNT 没有在第②步清零！
   CNT 继续从 10000 往上计数 → 到第③上升沿时 CNT = 20000
   硬件：CCR1 = CNT = 20000 ❌（错误！应该还是 10000）
   中断：count=1 → CNT=0（这时候才清零，但已经晚了）

④ 第 4 上升沿：
   硬件：CCR1 = CNT = 10000 ✅（又正确了）
   中断：count=2 → cycle = CCR1 → count=0

结果：CCR1 在 10000 和 20000 之间交替
     周期在 10ms 和 20ms 之间变化
```

### 2.3 为什么原始版本没问题

```
原始版本用 cycle 全局变量：
  → 只在 count=2 时才把 CCR1 写入 cycle
  → count=1 时即使 CCR1=20000，也不会写入 cycle
  → cycle 始终保存的是正确的 10000
  → 全局变量起到了"过滤错误值"的作用
```

## 三、优化方案（直接清零 + 直接读 CCR1）

### 3.1 核心思想

```
不需要全局变量！不需要 count！
每次上升沿只需：清除标志 + 清零 CNT

为什么可行？
  → 硬件捕获（CCR1 = CNT）先于中断处理（CNT = 0）
  → CCR1 已经保存了正确的周期值
  → 然后才清零 CNT，准备下一个周期
```

### 3.2 优化后代码

```c
// 不需要任何全局变量！

void TIM4_IRQHandler(void)
{
    if (TIM4->SR & TIM_SR_CC1IF)
    {
        TIM4->SR &= ~TIM_SR_CC1IF;    // 清除标志
        TIM4->CNT = 0;                  // 清零计数器
    }
}

double TIM4_GetPWMPeriod(void)
{
    return (double)TIM4->CCR1 / 1000.0;
}

double TIM4_GetPWMFrequency(void)
{
    if (TIM4->CCR1 == 0)
        return 0;
    return 1000000.0 / (double)TIM4->CCR1;
}
```

### 3.3 优化后执行流程

```
① 第 1 上升沿：
   硬件：CCR1 = CNT（快照，此时值无意义）
   中断：CNT = 0

② 第 2 上升沿：
   硬件：CCR1 = CNT = 10000（快照，正确！）
   中断：CNT = 0（准备下一周期）

③ 第 3 上升沿：
   硬件：CCR1 = CNT = 10000（快照，正确！）
   中断：CNT = 0

CCR1 始终保持 10000 ✅
```

## 四、事件 vs 中断 — 先后顺序

### 4.1 两条路径

```
上升沿到来
    │
    ├──→ 事件路径（硬件处理）
    │    │
    │    ├── 事件屏蔽寄存器 & 事件标志
    │    ↓
    │    硬件电路直接执行：CCR1 = CNT
    │    → 瞬间完成（1 个时钟周期）
    │
    └──→ 中断路径（软件处理）
         │
         ├── 中断屏蔽寄存器 & 中断标志
         ↓
         → NVIC 仲裁
         → CPU 响应
         → 保存现场
         → 查中断向量表
         → 调用中断服务函数
         → 执行代码：CNT = 0
         → 恢复现场
         → 延迟若干时钟周期
```

### 4.2 关键对比

| 对比项 | 事件处理 | 中断处理 |
|--------|:--------:|:--------:|
| 处理方式 | **硬件**自动 | **软件**（CPU 执行指令） |
| 执行速度 | **瞬间**（~1 周期） | **延迟**（若干周期） |
| 是否可控 | 不可控（硬件决定） | 可控（编写代码） |
| 本实验 | CCR1 = CNT（快照） | CNT = 0（清零） |

### 4.3 时序关系

```
时间轴：

上升沿到来
    │
    ├─[0]────→ 硬件事件：CCR1 = CNT = 10000  ← 瞬间完成
    │
    ├─[1~N]──→ 中断处理流程：
    │            NVIC 仲裁
    │            CPU 响应
    │            保存现场
    │            跳转中断服务函数
    │            执行：CNT = 0                ← 延迟 N 个周期
    │            恢复现场
    │
    └─[N+1]──→ 中断返回

关键：CCR1 = CNT（硬件）永远先于 CNT = 0（软件）
     这是硬件设计保证的，不会改变
```

---

## 五、为什么这个顺序至关重要

```
✅ 正确顺序（实际）：
  先：硬件捕获 CCR1 = CNT = 10000（保存正确值）
  后：软件清零 CNT = 0（准备下一周期）
  → CCR1 保持 10000 ✅

❌ 假设反过来（不可能发生）：
  先：软件清零 CNT = 0
  后：硬件捕获 CCR1 = CNT = 0（错误值！）
  → CCR1 = 0 ❌

结论：硬件事件处理永远快于中断软件处理
     这是硬件设计保证的物理特性
```

## 六、优化前后对比

| 对比项 | 优化前 | 优化后 |
|--------|--------|--------|
| 全局变量 | cycle + count | **无** |
| 中断函数 | 清标志 + count++ + 判断 + CNT=0 或读 CCR1 | 清标志 + CNT=0 |
| GetPeriod | return cycle/1000 | return CCR1/1000 |
| GetFrequency | return 1000000/cycle | return 1000000/CCR1 |
| 代码行数 | ~20 行 | **~5 行** |
| 逻辑 | 状态机（两次上升沿） | **每次上升沿都清零** |

---

## 七、理解要点总结

| 要点 | 说明 |
|------|------|
| **事件先于中断** | 硬件事件瞬间完成，中断软件延迟执行 |
| **CCR1 捕获先于 CNT 清零** | 硬件保证的时序 |
| **CCR1 保持不变** | 在下次捕获前不会被改变 |
| **每次上升沿只需清零 CNT** | 无需状态机，无需全局变量 |
| **直接读 CCR1 即可** | 无需缓存到全局变量 |
